FxSuite launcher — two alternative architectures

Both make the launcher a standing native app and shrink the browser's role, cutting copycat‑website risk. One backend serves both. Apps are still downloaded on demand and integrity‑checked before they run.

variant 1 · push

Server‑mediated launch

The web page talks only to the server; the server pushes a signed command to the agent.

  1. Browser → server. The page calls only the server: POST /launch (app, version). No launch URL ever reaches the machine.
  2. Server → agent. The server checks the request, RSA‑signs a launch command, and pushes it down the agent's standing WebSocket.
  3. Agent → app. The agent verifies the signature (and the app's SHA‑256) and launches it.
Why copycat‑resistant: the browser never contacts the agent, and a copycat site can neither reach the agent nor forge a server‑signed command.
PoC today: ws:// localhost, shared‑token agent auth, web session stubbed. Production: wss on 443, real login + session‑to‑agent pairing.
variant 2 · pull

Launcher‑native — the local launcher handles everything

No browser, no protocol handler, no push channel — so there is no web attack surface at all.

  1. Launcher → backend. The launcher requests the catalogue: GET /catalogue.
  2. Backend → launcher. Returns apps + versions + each jar's sha256. The launcher lists them in its own UI.
  3. You pick, launcher downloads. On Launch it fetches that version (GET /repo/…), verifies the SHA‑256 against the catalogue, and caches it.
  4. Launcher → app. Spawns the app as its own process, sharing the one JavaFX jar.
Why copycat‑proof: there is no local endpoint and no per‑app URL anywhere — no website has anything to target.
PoC today: catalogue over plain HTTP on localhost. Production: wss/HTTPS + a signed catalogue so the launcher trusts the versions and hashes.

Reach for variant 1 when…

  • you want a web front‑end to drive launches
  • the server must stay in the loop (entitlements, audit, remote kill)
  • selection lives on a page, not in the desktop app

Reach for variant 2 when…

  • you want zero web attack surface
  • the launcher should work with no browser open
  • a self‑contained desktop client is the whole experience
